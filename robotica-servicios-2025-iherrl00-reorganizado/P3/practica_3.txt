EJERCICIO 6: ROS2 BRIDGE

El ros_gz_bridge es fundamental para comunicar Gazebo con ROS 2, traduciendo
mensajes entre ambos sistemas.

6.1 CONCEPTO DEL BRIDGE
¿QUÉ ES EL BRIDGE?
- Traduce mensajes entre ROS 2 y Gazebo
- Permite comunicación bidireccional
- Mapea tipos de mensajes entre sistemas

¿POR QUÉ ES NECESARIO?
- Gazebo usa su propio sistema de mensajes
- ROS 2 tiene sus propios tipos de mensajes
- El bridge hace la conversión automática

6.2 SINTAXIS BÁSICA
COMANDO GENERAL:
ros2 run ros_gz_bridge parameter_bridge /TOPIC@ROS_MSG@GZ_MSG

EJEMPLO:
ros2 run ros_gz_bridge parameter_bridge /cube_camera/camera_info@sensor_msgs/msg/CameraInfo@ignition.msgs.CameraInfo

SÍMBOLOS DE DIRECCIÓN:
- @ : Bidireccional (ROS ↔ Gazebo)
- [ : Unidireccional Gazebo → ROS 2
- ] : Unidireccional ROS 2 → Gazebo

6.3 EJEMPLO: KEY PUBLISHER BRIDGE
Vamos a conectar el Key Publisher de Gazebo con ROS 2.

PASO 1: Obtener tipo de mensaje de Gazebo
ign topic -i -t /keyboard/keypress

Esto mostrará: ignition.msgs.Int32

PASO 2: Iniciar el bridge
ros2 run ros_gz_bridge parameter_bridge /keyboard/keypress@std_msgs/msg/Int32[ignition.msgs.Int32

NOTA EL SÍMBOLO [: 
Indica que los mensajes solo van de Gazebo a ROS 2.

PASO 3: Escuchar en ROS 2
En otra terminal:
ros2 topic echo /keyboard/keypress

RESULTADO ESPERADO:
Verás los códigos de teclas publicados en el topic de ROS 2 cuando
presiones teclas en Gazebo.

6.4 BRIDGE DE SENSORES
Para sensores es más conveniente usar un launch file:

LANZAR SIMULACIÓN:
ros2 launch gazebo_sensors cube_with_sensors.launch.py

LANZAR BRIDGE:
ros2 launch gazebo_sensors gazebo_bridge.launch.py

6.5 VERIFICAR TIPOS DE MENSAJE
Es importante conocer los tipos de mensaje para configurar el bridge:

CÁMARA INFO:
ign topic -i -t /cube_camera/camera_info
# Tipo: CameraInfo

IMAGEN RAW:
ign topic -i -t /cube_camera/image_raw
# Tipo: Image

NUBE DE PUNTOS:
ign topic -i -t /cube_depth/image_raw/points
# Tipo: PointCloudPacked

LASER SCAN:
ign topic -i -t /lidar
# Tipo: LaserScan

EJERCICIO 7: ROBOT DIFFERENTIAL DRIVE

Este ejemplo muestra cómo crear un robot móvil con tracción diferencial,
capaz de moverse en el entorno simulado.

7.1 CREAR EL PAQUETE
ros2 pkg create diff_drive_description --dependencies xacro
cd diff_drive_description
mkdir urdf launch
nano urdf/diff_drive.urdf.xacro
nano urdf/diff_drive_macro.xacro
nano launch/diff_drive.launch.py

¿POR QUÉ DOS ARCHIVOS XACRO?
- diff_drive.urdf.xacro: Archivo principal
- diff_drive_macro.xacro: Macros reutilizables (ruedas, etc.)

7.2 ESTRUCTURA DEL ROBOT
El robot differential drive típicamente tiene:

COMPONENTES:
- Base circular (link principal)
- 2 ruedas activas con continuous joints
- 2 ruedas pasivas (casters) con fixed joints
- Sensor LIDAR para navegación

¿QUÉ ES UN CONTINUOUS JOINT?
- Joint que puede rotar infinitamente
- No tiene límites de rotación
- Ideal para ruedas

7.3 PLUGIN DIFFERENTIAL DRIVE
Este plugin controla el movimiento del robot:

CONFIGURACIÓN EN XACRO:
<plugin filename="libignition-gazebo-diff-drive-system.so" 
        name="ignition::gazebo::systems::DiffDrive">
    <left_joint>left_wheel_joint</left_joint>
    <right_joint>right_wheel_joint</right_joint>
    <wheel_separation>0.3</wheel_separation>
    <wheel_radius>0.039</wheel_radius>
    <odom_publish_frequency>10</odom_publish_frequency>
    <topic>cmd_vel</topic>
    <robotBaseFrame>base_link</robotBaseFrame>
</plugin>

PARÁMETROS IMPORTANTES:
- left_joint/right_joint: Nombres de los joints de las ruedas
- wheel_separation: Distancia entre ruedas (metros)
- wheel_radius: Radio de las ruedas (metros)
- odom_publish_frequency: Hz de publicación de odometría
- topic: Topic para comandos de velocidad (cmd_vel)

7.4 COMPILAR Y LANZAR
COMPILAR:
cd ~/ros2_ws
colcon build --symlink-install
source install/setup.bash

LANZAR:
ros2 launch diff_drive_description diff_drive.launch.py

7.5 CONTROLAR EL ROBOT
Existen varias formas de enviar comandos de velocidad:

OPCIÓN A: Usando RQT (Recomendado)
rqt

Luego seleccionar: Plugins → Robot Tools → Robot Steering

OPCIÓN B: Por línea de comandos
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist "{linear: {x: 0.5}, angular: {z: 0.5}}"

EXPLICACIÓN DEL MENSAJE:
- linear.x: Velocidad lineal (m/s) hacia adelante/atrás
- angular.z: Velocidad angular (rad/s) rotación

7.6 TOPICS DEL BRIDGE
El differential drive usa estos topics:

COMANDO DE VELOCIDAD:
- Topic: /cmd_vel
- Dirección: ROS 2 → Gazebo
- Tipo: geometry_msgs/msg/Twist

ODOMETRÍA:
- Topic: /model/diff_drive/odometry
- Dirección: Gazebo → ROS 2
- Tipo: nav_msgs/msg/Odometry

¿QUÉ ES ODOMETRÍA?
- Estimación de posición del robot
- Basada en rotación de ruedas
- Incluye pose y velocidad

8.4 ESTRUCTURA DE UN PLUGIN EN C++
CÓDIGO BÁSICO:
#include <gz/plugin/Register.hh>
#include <gz/sim/System.hh>

namespace mi_plugin {
    class MiPlugin : 
        public gz::sim::System,
        public gz::sim::ISystemPostUpdate
    {
        public: void PostUpdate(
            const gz::sim::UpdateInfo &_info,
            const gz::sim::EntityComponentManager &_ecm) override;
    };
}

IGNITION_ADD_PLUGIN(
    mi_plugin::MiPlugin, 
    gz::sim::System,
    mi_plugin::MiPlugin::ISystemPostUpdate
)

INTERFACES DISPONIBLES:
- ISystemConfigure: Inicialización
- ISystemPreUpdate: Antes de cada paso
- ISystemUpdate: Durante cada paso
- ISystemPostUpdate: Después de cada paso

EJERCICIO 9: FORMATOS DE ARCHIVO EN ROS/GAZEBO

Es importante entender los diferentes formatos de archivo utilizados
en robótica para elegir el apropiado para cada tarea.

9.1 URDF (UNIFIED ROBOT DESCRIPTION FORMAT)
¿QUÉ ES?
- Formato XML para describir robots en ROS
- Estándar de facto en ROS
- Relativamente simple pero limitado

COMPONENTES:
- Links: Partes físicas del robot
- Joints: Conexiones entre links
- Visual: Geometría para visualización
- Collision: Geometría para colisiones
- Inertial: Propiedades físicas

LIMITACIONES:
- No puede describir mundos completos
- Sensores limitados
- No soporta física compleja

USO TÍPICO:
- Descripción estática de robots
- Visualización en RViz
- Cinemática básica

9.2 SDF (SIMULATION DESCRIPTION FORMAT)
¿QUÉ ES?
- Formato XML agnóstico a ROS
- Más completo y flexible que URDF
- Estándar para Gazebo

VENTAJAS SOBRE URDF:
- Puede describir mundos completos
- Sensores avanzados
- Física detallada
- Plugins integrados
- Múltiples modelos en un archivo

USO TÍPICO:
- Simulaciones complejas en Gazebo
- Mundos con múltiples robots
- Física avanzada

COMPATIBILIDAD:
- URDF puede convertirse a SDF
- SDF no siempre puede convertirse a URDF

9.3 XACRO (XML MACROS)
¿QUÉ ES?
- Extensión de URDF/XML
- Añade programabilidad a XML
- Reduce repetición de código

CARACTERÍSTICAS:
- Macros: Bloques de código reutilizables
- Variables: Parámetros configurables
- Lógica condicional: if/unless
- Operaciones matemáticas
- Inclusión de archivos

EJEMPLO DE MACRO:
<xacro:macro name="rueda" params="nombre radio">
  <link name="${nombre}">
    <visual>
      <geometry>
        <cylinder radius="${radio}" length="0.05"/>
      </geometry>
    </visual>
  </link>
</xacro:macro>

PROCESAMIENTO:
xacro file.xacro > output.urdf

USO TÍPICO:
- Robots con partes repetitivas (ruedas)
- Configuraciones parametrizables
- Proyectos grandes modulares

9.4 DAE (COLLADA DIGITAL ASSET EXCHANGE)
¿QUÉ ES?
- Formato 3D basado en XML
- Estándar de intercambio de modelos 3D
- Ampliamente soportado

CONTENIDO:
- Geometría 3D (meshes)
- Texturas y materiales
- Colores y shaders
- Animaciones (opcional)
- Cámaras y luces

USO EN ROS/GAZEBO:
- Visualización de robots con modelos complejos
- Meshes personalizados
- Apariencia realista

HERRAMIENTAS COMPATIBLES:
- Blender (open source)
- Maya
- 3ds Max
- SketchUp

CONVERSIÓN:
Muchas herramientas 3D pueden exportar a DAE.

EJERCICIO 10: LAUNCH FILES EN ROS 2

Los launch files permiten iniciar múltiples nodos y configurar parámetros
de forma automatizada.

10.1 TIPOS DE LAUNCH FILES
ROS 2 soporta tres formatos:

1. PYTHON (.py)
VENTAJAS:
- Máxima flexibilidad
- Lógica avanzada (if/else, loops)
- Integración con código Python

DESVENTAJAS:
- Más verboso
- Requiere conocimientos de Python

2. XML (.xml)
VENTAJAS:
- Sintaxis simple
- Compatible con ROS 1
- Fácil de leer

DESVENTAJAS:
- Menos flexible
- No permite lógica compleja

3. YAML (.yaml)
VENTAJAS:
- Muy compacto
- Fácil de escribir
- Bueno para configuraciones

DESVENTAJAS:
- Funcionalidad limitada
- No todo está implementado aún

10.2 ESTRUCTURA BÁSICA EN PYTHON
PLANTILLA:
from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
    return LaunchDescription([
        Node(
            package='mi_paquete',
            executable='mi_nodo',
            name='mi_nodo_nombre',
            output='screen',
            parameters=[{'param': value}]
        )
    ])

COMPONENTES:
- LaunchDescription: Contenedor principal
- Node: Representa un nodo a lanzar
- package: Nombre del paquete ROS
- executable: Nombre del ejecutable
- name: Nombre del nodo en runtime
- output: Dónde mostrar la salida
- parameters: Diccionario de parámetros

10.3 LANZAR UN LAUNCH FILE
SINTAXIS:
ros2 launch paquete archivo.launch.py

EJEMPLO:
ros2 launch diff_drive_description diff_drive.launch.py

BÚSQUEDA:
ros2 launch busca automáticamente en:
- share/paquete/launch/

10.4 ELEMENTOS AVANZADOS
INCLUIR OTROS LAUNCH FILES:
from launch.actions import IncludeLaunchDescription

IncludeLaunchDescription(
    PythonLaunchDescriptionSource('otro.launch.py')
)

ARGUMENTOS:
from launch.actions import DeclareLaunchArgument

DeclareLaunchArgument(
    'use_sim_time',
    default_value='true'
)

CONDICIONALES:
from launch.conditions import IfCondition

Node(
    ...,
    condition=IfCondition(LaunchConfiguration('use_camera'))
)

EJERCICIO 11: CONCEPTOS CLAVE DE GAZEBO Y ROS 2

11.1 GAZEBO TOPICS
¿QUÉ SON?
- Sistema de comunicación interno de Gazebo
- Similar a ROS topics pero independiente
- Usa el protocolo Ignition Transport

DIFERENCIAS CON ROS TOPICS:
- No son automáticamente accesibles desde ROS
- Requieren bridge para interoperar
- Tipos de mensaje diferentes

COMANDOS ÚTILES:
ign topic -l                    # Listar topics
ign topic -i -t /topic          # Info del topic
ign topic -e -t /topic          # Escuchar topic

11.2 ROS_GZ_BRIDGE
¿QUÉ ES?
- Puente de comunicación entre ROS 2 y Gazebo
- Traduce tipos de mensajes
- Permite control bidireccional

FUNCIONAMIENTO:
1. Escucha topics de Gazebo
2. Traduce mensajes al formato ROS
3. Publica en topics ROS
4. Y viceversa

LIMITACIONES:
- No todos los tipos de mensaje son compatibles
- Puede introducir latencia mínima
- Requiere configuración explícita

11.3 PLUGINS DE GAZEBO
¿QUÉ SON?
- Componentes que añaden funcionalidad
- Se cargan dinámicamente
- Pueden modificar la simulación

TIPOS:
- World: Afectan al mundo completo
- Model: Afectan a un modelo específico
- Sensor: Procesan datos de sensores
- System: Funcionalidad general

VENTAJA EN MODERN GAZEBO:
- Un plugin universal para todos los sensores
- Configuración más simple
- Mejor rendimiento

11.4 ROBOT_STATE_PUBLISHER
¿QUÉ ES?
- Nodo que publica transformadas (TF) del robot
- Lee la descripción del robot
- Calcula cinemática directa

FUNCIONAMIENTO:
1. Lee URDF del topic /robot_description
2. Calcula posición de todos los links
3. Publica transformadas en /tf

¿POR QUÉ ES NECESARIO?
- RViz necesita las transformadas para visualizar
- Otros nodos necesitan saber dónde están las partes
- Fundamental para navegación y manipulación

LANZAMIENTO TÍPICO:
Node(
    package='robot_state_publisher',
    executable='robot_state_publisher',
    parameters=[{'robot_description': robot_desc}]
)

EJERCICIO 12: DEBUGGING Y HERRAMIENTAS

12.1 VERIFICAR TOPICS ROS 2
LISTAR TODOS LOS TOPICS:
ros2 topic list

VER MENSAJES EN TIEMPO REAL:
ros2 topic echo /topic_name

EJEMPLO:
ros2 topic echo /cube_camera/camera_info

INFORMACIÓN DEL TOPIC:
ros2 topic info /topic_name

Muestra:
- Tipo de mensaje
- Número de publishers
- Número de subscribers

FRECUENCIA DE PUBLICACIÓN:
ros2 topic hz /topic_name

12.2 VERIFICAR NODOS ROS 2
LISTAR NODOS ACTIVOS:
ros2 node list

INFORMACIÓN DE UN NODO:
ros2 node info /node_name

Muestra:
- Subscribers
- Publishers
- Services
- Actions

12.3 VERIFICAR TRANSFORMADAS (TF)
GENERAR GRÁFICO DE TRANSFORMADAS:
ros2 run tf2_tools view_frames

Genera un PDF con el árbol de transformadas.

VER TRANSFORMADA ENTRE DOS FRAMES:
ros2 run tf2_ros tf2_echo frame1 frame2

EJEMPLO:
ros2 run tf2_ros tf2_echo base_link lidar_link

12.4 LOGS DE GAZEBO
AUMENTAR VERBOSIDAD:
ign gazebo -v 3 world.sdf  # Nivel 3
ign gazebo -v 4 world.sdf  # Nivel 4 (máximo)

NIVELES:
- 0: Silencioso
- 1: Errores
- 2: Warnings
- 3: Info
- 4: Debug

EJERCICIO 13: ERRORES COMUNES Y SOLUCIONES

13.1 PLUGIN NO ENCONTRADO
SÍNTOMA:
Error: "Failed to load plugin"

SOLUCIÓN:
Exportar la ruta de plugins:
export GZ_SIM_SYSTEM_PLUGIN_PATH=/ruta/a/plugins

HACER PERMANENTE:
Añadir al ~/.bashrc:
echo 'export GZ_SIM_SYSTEM_PLUGIN_PATH=/ruta/a/plugins' >> ~/.bashrc

13.2 BRIDGE NO FUNCIONA
SÍNTOMA:
Los topics no aparecen en ROS 2

DIAGNÓSTICO:
1. Verificar tipo de mensaje en Gazebo:
   ign topic -i -t /topic_name

2. Verificar tipo en ROS 2:
   ros2 interface show tipo/mensaje

3. Confirmar compatibilidad entre tipos

SOLUCIÓN:
Usar los tipos de mensaje correctos en el bridge.

13.3 ROBOT CAE INFINITAMENTE
SÍNTOMA:
El robot desaparece hacia abajo en la simulación

CAUSA:
No hay plano de suelo en el mundo

SOLUCIÓN A:
Usar un mundo que incluya suelo:
ign gazebo empty.sdf

SOLUCIÓN B:
Añadir ground plane manualmente en el archivo SDF del mundo.

13.4 COMPILACIÓN FALLA
SÍNTOMA:
Errores durante colcon build

SOLUCIÓN 1: Limpiar workspace
rm -rf build install log
colcon build --symlink-install
source install/setup.bash

SOLUCIÓN 2: Verificar dependencias
rosdep install --from-paths src -y --ignore-src

SOLUCIÓN 3: Compilar paquete individual
colcon build --packages-select nombre_paquete
